## What Is Spening?

Spening is a single-user, self-hosted web app that connects to the
[Plaid API](https://plaid.com/docs/) to pull in bank and credit card transactions, stores them in
PostgreSQL, and presents reports and transaction tables in a browser UI.

**There is no login/authentication** вЂ” it is designed to run on a private home network or NAS.

### Feature Set
- **Link accounts** вЂ” connect bank/credit card accounts via Plaid Link (OAuth flow, runs in-browser)
- **Sync transactions** вЂ” incremental sync using Plaid's `/transactions/sync` endpoint (cursor-based)
- **Fetch transactions** вЂ” full re-fetch for a configured calendar year via `/transactions/get`
- **Transaction table** вЂ” filterable/sortable table with CSV export
- **Reports** вЂ” five report views: monthly totals, by-category totals, monthГ—category pivot,
  top merchants, largest transactions; each exportable as CSV
- **Settings** вЂ” configure active year and Plaid environment (sandbox/development/production)
- **Remove accounts** вЂ” revoke Plaid access token and delete local data

### Plaid Concepts (critical background)
- A **Plaid Item** represents one bank login. One Item can have multiple accounts (e.g. checking +
  savings at the same bank). All accounts under an Item share one `access_token`.
- An `access_token` is obtained by: creating a `link_token` server-side в†’ user completes Plaid Link
  in the browser в†’ browser receives a `public_token` в†’ server exchanges `public_token` for
  `access_token` via `/item/public_token/exchange`.
- **Sync** uses a cursor per Item. On first call, omit the cursor. Plaid returns `added`,
  `modified`, and `removed` transaction arrays plus `has_more` and `next_cursor`. Loop until
  `has_more` is false. Persist the final `next_cursor` to use next time.
- **Amounts**: Plaid returns positive amounts for money leaving the account (expenses) and negative
  for money coming in (income/refunds). The app treats positive amounts as expenses.
- **personal_finance_category**: request `include_personal_finance_category: true` in options.
  Use the `primary` field as the category string. Fall back to the legacy `category` array
  (joined with ` > `) if not present.

## Tech Stack

| Layer | Technology |
|---|---|
| Framework | ASP.NET Core 9 вЂ” Blazor Server + API Controllers in one project |
| ORM | Entity Framework Core 9 with Npgsql provider |
| Database | PostgreSQL 16 |
| UI component library | None вЂ” plain Blazor with minimal CSS |
| Containerization | Docker (multi-stage Dockerfile) + Docker Compose |
| CI/CD | GitHub Actions в†’ Docker Hub |
| Target runtime | Self-hosted homelab / NAS |

**Why a single ASP.NET Core project?** Blazor Server pages and API controllers coexist in one
project. There are no inter-service HTTP calls вЂ” Blazor pages call services directly, and the API
controllers exist for operations that need to be called programmatically or tested with curl.
One project = one image = simpler homelab operation.

## Database Schema

### Table: `linked_accounts`
| Column | Type | Notes |
|---|---|---|
| `id` | `uuid` PK | generated by app (`Guid.NewGuid()`) |
| `access_token` | `text` NOT NULL | AES-256-GCM encrypted before storage |
| `account_id` | `text` NOT NULL UNIQUE | Plaid's account_id |
| `mask` | `text` | last 4 digits, may be empty |
| `name` | `text` NOT NULL | account name from Plaid |
| `subtype` | `text` | e.g. `checking`, `savings`, `credit card` |
| `institution` | `text` | institution display name |
| `sync_cursor` | `text` | nullable; updated after every sync |
| `created_at` | `timestamptz` | set on insert |

### Table: `transactions`
| Column | Type | Notes |
|---|---|---|
| `transaction_id` | `text` PK | Plaid's stable transaction_id |
| `account_id` | `text` NOT NULL | Plaid account_id (not a FK вЂ” accounts may be removed) |
| `date` | `date` NOT NULL | |
| `name` | `text` NOT NULL | merchant / description |
| `amount` | `numeric(12,2)` NOT NULL | positive = expense |
| `category` | `text` | from personal_finance_category.primary |
| `created_at` | `timestamptz` | |
| `updated_at` | `timestamptz` | touch on every upsert |

### Table: `app_settings`
| Column | Type | Notes |
|---|---|---|
| `key` | `text` PK | |
| `value` | `text` NOT NULL | |

**Default settings rows** (insert on first run if absent):

| key | default value |
|---|---|
| `plaid_environment` | `sandbox` |
| `output_year` | current calendar year |

## Environment Variables

| Variable | Required | Description |
|---|---|---|
| `PLAID_CLIENT_ID` | Yes | Plaid dashboard client ID |
| `PLAID_SANDBOX_SECRET` | Yes | Plaid sandbox secret |
| `PLAID_PRODUCTION_SECRET` | No | Plaid production secret |
| `ENCRYPTION_KEY` | Yes | Base64-encoded 32-byte key for AES-256 access token encryption |
| `DB_PASSWORD` | Yes | PostgreSQL password |
| `ConnectionStrings__Default` | Yes (set by compose) | Full Npgsql connection string |

Secrets **never** go in `appsettings.json` or the database. They are injected at runtime via
Docker Compose environment variables sourced from a `.env` file.

## How to Resume After a Token Interruption

If you run out of tokens mid-phase:

1. On restart, read `PLAN.md` (this file) first to re-establish full context.
2. Read the phase file you were working on.
3. Each phase file has a **Progress Tracker** section at the top listing every task as a checkbox.
   Determine which tasks are complete by inspecting the existing files in the repository.
4. Resume from the first incomplete task. Do not re-do completed tasks.
5. When a phase is fully complete, all checkboxes in its Progress Tracker should be checkable.

**How to determine task completion without running the app:**
- A model/entity task is done if the corresponding `.cs` file exists with the correct properties.
- A migration task is done if `Data/Migrations/` contains at least one file.
- A service task is done if the service class exists and is registered in `Program.cs`.
- A controller task is done if the controller class exists with the expected action methods.
- A Blazor page task is done if the `.razor` file exists with its `@page` directive.
- A Docker task is done if `Dockerfile` / `docker-compose.yml` exist and are non-trivial.
- A CI task is done if `.github/workflows/docker-publish.yml` exists.

## Coding Conventions

- **Nullable reference types** enabled (`<Nullable>enable</Nullable>`)
- **Implicit usings** enabled
- All async methods use `async/await`; no `.GetAwaiter().GetResult()` in service/controller code
- `AppDbContext` is registered as scoped; inject via constructor
- Plaid HTTP calls use a single `HttpClient` instance per service (constructor-injected or `IHttpClientFactory`)
- Error responses from Plaid: throw `InvalidOperationException` with the Plaid error body included
- All controller actions return `IActionResult` or `ActionResult<T>`
- Blazor pages handle loading/error state explicitly (bool `_loading`, `string? _error` fields)
- `ENCRYPTION_KEY` env var is read once at startup in `Program.cs` and registered as a singleton
  `EncryptionService`

## Plaid API Reference (endpoints used)

| Endpoint | Used for |
|---|---|
| `POST /link/token/create` | Generate link_token to initialise Plaid Link in browser |
| `POST /item/public_token/exchange` | Exchange browser's public_token for server-side access_token |
| `POST /accounts/get` | Fetch accounts for a given access_token |
| `POST /transactions/sync` | Incremental cursor-based transaction sync |
| `POST /transactions/get` | Full date-range transaction fetch |
| `POST /institutions/get_by_id` | Resolve institution_id в†’ display name |
| `POST /item/remove` | Revoke access token server-side |

Base URLs:
- Sandbox: `https://sandbox.plaid.com`
- Development: `https://development.plaid.com`
- Production: `https://production.plaid.com`

All requests are `POST` with `Content-Type: application/json`. All request bodies include
`client_id` and `secret` fields.
